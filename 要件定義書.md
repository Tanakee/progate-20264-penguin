# 要件定義書：ファースト肯定ペンギン

## 1. プロジェクト概要

- **コンセプト**: 最初に拍手をした観客（ファーストペンギン）を検知してARでペンギン化し、その肯定アクションをAWSでデータ化・分析する。
- **コアバリュー**: 勇気ある肯定をリアルタイムに視覚化し、その熱量を客観的なデータとして蓄積・提供する。

---

## 2. 機能要件

### 2.1 リアルタイム検知・AR合成（エッジ処理）

- **初動検知**: ローカルPCのマイクで音圧を監視し、拍手音を検知する。音の検知はトリガーとして使用し、人物特定は映像解析で行う。
- **人物特定**: 拍手検知タイミングの映像フレームに対し、MediaPipe Hands を用いて**最初に手を叩く動作（両手のランドマークが急接近する動き）を開始した人物の座標を特定**する。複数人が同時に動作を開始した場合は、映像の中央に最も近い人物を優先して選択する。
- **AR合成**: OpenCV等を使用し、特定された人物にペンギンのARアバターをリアルタイムに重ね合わせ、会場モニターへ直接出力する。
- **二重保存**: AR合成処理と並行して、**合成前の原画像**と**合成後の画像**をそれぞれ別途保存する（詳細は2.2参照）。

### 2.2 AWSデータ連携（リアルタイム同期・保存）

- **判定フラグ送信**: 拍手検知の結果（時間・座標等）を AWS AppSync を介してダッシュボード等のクライアントへ即時配信する。
- **証拠写真保存（二種類）**:
  - **原画像**: Rekognition による顔・感情分析に使用するため、AR合成前のスナップショットを `s3://bucket/raw/` へ自動アップロードする。
  - **合成画像**: 会場演出・記録用として、AR合成後のスナップショットを `s3://bucket/composed/` へ自動アップロードする。
- **データフロー**: エッジ（ローカルPC）が直接S3へ画像をアップロードし、S3イベント通知をトリガーとしてAWS Lambdaが起動。LambdaがRekognitionでの解析・Timestreamへの書き込みを実行する。AppSyncはフロントエンドへのリアルタイム結果配信専用とする。

### 2.3 肯定アナリティクス（AI分析・可視化）

- **感情・属性分析**: S3に保存された**原画像**を Amazon Rekognition で解析し、拍手した人の笑顔度（肯定の強さ）や属性を数値化する。
- **時系列蓄積**: 全ての肯定データを Amazon Timestream に保存し、イベント全体の盛り上がりを時系列で管理する。
- **ダッシュボード**: Amazon QuickSight を活用し、発表終了後の「最大肯定指数」や「盛り上がりピーク」をグラフとして可視化する。

---

## 3. システム構成 (AWS Architecture)

| コンポーネント | 使用サービス | 役割 |
|---|---|---|
| データ配信 | AWS AppSync | 判定結果のフロントエンドへのリアルタイム配信 |
| ストレージ | Amazon S3 | 原画像（分析用）・合成画像（記録用）の保存 |
| AI解析 | Amazon Rekognition | 原画像から笑顔（肯定の度合い）や属性を分析 |
| データベース | Amazon Timestream | 拍手発生タイミングと分析結果の時系列保存 |
| 分析可視化 | Amazon QuickSight | 会場の熱狂度を可視化するダッシュボードの提供 |
| ロジック実行 | AWS Lambda | S3への画像保存をトリガーとした解析ジョブの実行 |

### データフロー図

```
[ローカルPC]
  マイク（音圧監視）
    ↓ 拍手音検知
  カメラ映像フレーム → MediaPipe Hands で最初の拍手動作者を特定
    ↓
  ┌─────────────────────────────────────┐
  │  原画像 → S3 (raw/)                 │  ← Rekognition分析用
  │  合成画像 → S3 (composed/)           │  ← 演出・記録用
  │  判定メタデータ → AppSync → クライアント │  ← リアルタイム配信
  └─────────────────────────────────────┘
    ↓ S3イベント通知
  Lambda
    ├─ Rekognition（原画像を解析）
    └─ Timestream（分析結果を時系列保存）
    ↓
  QuickSight（ダッシュボード可視化）
```

---

## 4. 非機能要件

- **超低遅延**: 映像合成はローカルで行うことで遅延を最小化する。目標値として、拍手検知からAR表示までを **100ms以内** とする。
- **データ整合性**: AWS AppSyncにより、複数の管理端末や分析画面で常に最新の肯定状況が共有されること。
- **スケーラビリティ**: 複数の発表者が交代するハッカソンの運用に耐えうるサーバーレス設計とする。

---

## 5. エッジケース定義

| ケース | 対処方針 |
|---|---|
| 複数人が同時に拍手を開始した場合 | 映像中央に最も近い人物を優先選択する |
| 拍手音を検知したが映像で手の動作が確認できない場合 | 誤検知として無視し、次の検知を待つ |
| AR合成中に対象人物が画面外に出た場合 | 最後に特定された座標を保持し、一定時間後にAR表示を終了する |
